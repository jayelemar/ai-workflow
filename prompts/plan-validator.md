# Plan Validation (State-Machine Driven)

This prompt validates a plan generated by `/plan`.

It does NOT generate plans.

---

## Purpose

Ensure the plan is:

* aligned with spec (if exists)
* complete
* executable
* free of assumptions

---

## Instruction Loading

Read:

* `.codex/AGENTS.md`
* `.ai/instructions/shared/workflow-state.md`
* the plan file
* `.ai/specs/<feature>.spec.md` (if exists)
* relevant codebase files named by the spec or plan when contract, shape, rendering, or file-scope questions must be resolved from existing implementation evidence

## Spec Alignment (MANDATORY)

If a spec exists:

* every behavior in spec MUST be implemented in the plan
* NO additional behavior should be introduced
* NO behavior should be missing

If mismatch exists:

→ mark as CRITICAL

## Codebase Contract Resolution (MANDATORY)

Before classifying any issue as spec-origin, use existing codebase evidence from the spec-scoped or plan-scoped files to determine whether the issue is actually:

* a plan overreach that invents behavior beyond the spec
* a plan omission that failed to include behavior already required by the spec
* an existing contract/type/rendering path that the plan can reuse without needing a new product decision

Use the codebase as a contract source only when all of the following are true:

* the relevant files already exist
* the contract or behavior is already implemented or typed there
* aligning the plan to that contract does NOT introduce new product behavior beyond the spec

Classify as a plan issue, NOT a spec issue, when any of the following are true:

* the plan invents a new field shape, ordering rule, confidence rule, logging rule, rendering rule, or file scope that the spec did not require
* the plan can be corrected by removing invented behavior
* the plan can be corrected by reusing an existing codebase contract/type/rendering path in spec-scoped files
* the spec already defines the behavior, but the plan omitted it
* the plan adds out-of-scope files or tests not named or implied by the spec
* the spec adds a new section to an existing document/API surface and the plan reuses an existing sibling contract for equivalent items already present in that same surface
* the plan includes a supporting type/contract file that must change only to carry a spec-required shape through an already in-scope owner file

Rules:

* Existing codebase contracts SHOULD be preferred over escalating to user decisions when they allow the plan to stay within the spec.
* Do NOT call a data shape/API contract "unclear" if the existing spec-scoped codebase already defines a compatible contract the plan can reuse.
* Do NOT call a sibling-contract reuse choice a spec gap when the spec adds a new section to an existing document/API and the reused contract already represents the same kind of item in that surface.
* Do NOT call out-of-scope plan additions a spec gap; they are plan defects.
* Supporting files directly required to implement behavior in a spec-named owner file may appear in the plan when they do not expand behavior beyond the spec.
* Only treat an issue as spec-origin if the spec plus relevant existing codebase still leave multiple valid product behaviors or no reusable contract to anchor the plan.

## Spec Completeness Check (MANDATORY)

If a spec exists:

Check for:

* undefined behaviors
* vague rules
* missing edge cases

For every issue that originates from the spec itself, classify it before routing:

### Spec Issue Classification

`MINOR SPEC REPAIR` applies ONLY to:

* typos, formatting, heading/list consistency
* duplicated or conflicting wording where one side is clearly accidental
* terminology normalization
* making behavior explicit when it is already unambiguously defined elsewhere in the same spec

`MAJOR SPEC DECISION REQUIRED` applies to:

* new behavior
* changed business logic
* missing product choice
* unclear data shape/API contract
* unclear edge-case behavior
* conflicting valid interpretations
* anything that requires user authority

Rules:

* Run the Codebase Contract Resolution check before using this classification.
* If the issue cannot be classified with confidence as `MINOR SPEC REPAIR`, classify it as `MAJOR SPEC DECISION REQUIRED`.
* `MINOR SPEC REPAIR` MUST NOT introduce behavior that is not already decided in the existing spec.
* `MINOR SPEC REPAIR` MUST identify the exact spec file and exact section(s) that `fix-plan` is allowed to repair.
* `MAJOR SPEC DECISION REQUIRED` MUST output `STOP`, state the required user decision, and must not transition to `fix-plan`.

`MAJOR SPEC DECISION REQUIRED` does NOT apply when:

* the plan is inventing behavior that can simply be removed
* the codebase already provides a compatible contract/type/rendering path the plan can adopt
* the codebase already provides an equivalent sibling item contract that can be reused for the new spec-required section
* a supporting type/contract file must be updated only to carry the already-decided spec behavior through an in-scope owner file
* the problem is plan file scope, test scope, or implementation detail overreach rather than undecided behavior
* the spec requirement exists and the plan merely failed to include it

If minor spec repairs are detected:

→ keep plan in `draft`
→ set `Next Action` to `fix-plan`
→ append validation history
→ list the exact spec file and section(s) that `fix-plan` is allowed to repair

If major spec decisions are detected:

→ output `STOP`
→ state the required user decision
→ do not update the plan state
→ do not transition to `fix-plan`

## Behavior Integrity

Check:

* no inferred business logic
* no speculative rules
* no hidden assumptions

If found:

→ mark as CRITICAL

---

## Coverage Check

Ensure the plan includes:

* all required modules/files
* all logic paths
* validation steps
* edge cases (from spec)

If missing:

→ mark as WARNING or CRITICAL

---

## Phase Quality

Each phase must:

* have a clear objective
* map to specific files
* be independently executable

If unclear:

→ mark as WARNING

---

## Execution Readiness (MANDATORY)

Ensure:

* steps are actionable
* no ambiguity in implementation
* dependencies are defined

If not executable:

→ mark as CRITICAL

---

## Validation Quality

Ensure:

* test steps exist
* expected results are defined
* regression checks included

If weak:

→ mark as WARNING

---

## Validation History (MANDATORY)

Append validation results to:

## Validation History

Rules:

* MUST append a new entry for every validation run
* MUST NOT overwrite previous entries
* Validation versions MUST be sequential
* MUST create `.ai/artifacts/<plan-name>/events/validation-vX.md` before updating the plan
* The validation artifact MUST contain detailed critical issues, warnings, spec repair classifications, allowed spec repairs, recommendations, and evidence
* The plan entry MUST contain only `Summary`, `Result`, and `Evidence`
* The plan entry MUST stay under 512 bytes
